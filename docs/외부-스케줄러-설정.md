# 외부 스케줄러로 정시 알림 보내기 (cron-job.org)

## 왜 필요한가
GitHub Actions의 `schedule:` cron은 **정시 실행을 보장하지 않습니다**(공식적으로 best-effort).
혼잡 시 수십 분~수 시간 지연됩니다. 실제로 07:00 알림이 08:11, 17:45 알림이 20:17에 온 적이 있습니다.

그래서 **외부 무료 cron 서비스가 정시에 GitHub API(`workflow_dispatch`)를 호출**해
워크플로를 깨우도록 바꿨습니다. 외부 서비스는 GitHub의 작업 큐와 무관하게 정시에 동작하므로
지연이 거의 사라집니다.

> 워크플로 두 개(`daily.yml`, `evening.yml`)의 GitHub `schedule:` 은 중복 발송 방지를 위해
> 주석 처리(비활성화)했고, `workflow_dispatch` 만 남겨 외부 트리거로 실행합니다.

---

## 1단계: GitHub 토큰(PAT) 발급
외부 서비스가 워크플로를 실행하려면 권한 토큰이 필요합니다.

1. GitHub 로그인 → 우측 상단 프로필 → **Settings**
2. 좌측 맨 아래 **Developer settings** → **Personal access tokens** → **Fine-grained tokens** → **Generate new token**
3. 설정값
   - **Token name**: `calendar-bot-dispatch` (아무 이름)
   - **Expiration**: 1년 또는 No expiration(만료 시 갱신 필요)
   - **Repository access**: *Only select repositories* → **`nova-law-legal/legal`** 선택
   - **Permissions** → **Repository permissions** → **Actions** 항목을 **Read and write** 로 설정
4. **Generate token** → 나오는 토큰 문자열(`github_pat_...`)을 복사해 안전한 곳에 보관
   (이 화면을 벗어나면 다시 볼 수 없습니다.)

---

## 2단계: cron-job.org에 작업 2개 등록
https://cron-job.org 가입(무료) 후 **Create cronjob** 으로 아래 두 개를 만듭니다.

### 공통 설정 (두 작업 모두 동일)
- **Request method**: `POST`
- **Headers** (3개 추가)
  | Key | Value |
  |---|---|
  | `Accept` | `application/vnd.github+json` |
  | `Authorization` | `Bearer github_pat_여기에_복사한_토큰` |
  | `X-GitHub-Api-Version` | `2022-11-28` |
- **Request body** (JSON):
  ```json
  {"ref":"main"}
  ```
- **Time zone**: `Asia/Seoul (KST)` 로 지정 (중요 — UTC로 두면 시간이 9시간 어긋남)

### 작업 ① 오전 알림 (오늘 일정 전체)
- **URL**:
  ```
  https://api.github.com/repos/nova-law-legal/legal/actions/workflows/daily.yml/dispatches
  ```
- **Schedule**: 매일 **07:00** (KST)

### 작업 ② 오후 알림 (내일 일정, 팀별)
- **URL**:
  ```
  https://api.github.com/repos/nova-law-legal/legal/actions/workflows/evening.yml/dispatches
  ```
- **Schedule**: 매일 **17:30** (KST)

## 3단계: 동작 확인
1. cron-job.org에서 방금 만든 작업의 **Run now**(지금 실행) 버튼을 눌러 테스트.
2. 성공 시 응답 코드 **204 No Content** 가 떠야 정상입니다.
   (204 = "잘 받았고 워크플로를 시작함". 빈 응답이 정상)
3. GitHub 저장소 → **Actions** 탭에서 해당 워크플로가 방금 실행됐는지 확인.
4. 디스코드 채널에 알림이 도착했는지 확인.

### 응답 코드로 문제 진단
- **204**: 정상.
- **401 / 403**: 토큰이 틀렸거나 Actions 권한(Read and write)이 없음 → 1단계 재확인.
- **404**: URL의 저장소명/워크플로 파일명이 틀렸거나, 토큰이 그 저장소에 접근 권한 없음.
- **422**: body의 `ref`(브랜치)가 틀림 → 기본 브랜치가 `main` 이 맞는지 확인.

## 참고: 더 안정적으로 하려면 (선택)
- 외부 서비스 자체가 죽는 경우까지 대비하려면, 각 워크플로의 주석 처리된 `schedule:` 을 다시
  살려 **백업**으로 둘 수 있습니다. 단 이 경우 외부 트리거와 GitHub cron이 둘 다 실행돼
  **중복 발송**될 수 있으므로, 그때는 "오늘 이미 보냈으면 건너뛰기" 같은 중복 방지 로직이 필요합니다.
  (현재는 외부 트리거 단독 운영이라 중복 걱정이 없습니다.)
